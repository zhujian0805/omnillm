## 1. What This Project Does

**OmniLLM** is an enterprise-ready multi-provider LLM gateway. It exposes a single API surface compatible with both **OpenAI** and **Anthropic** endpoints while routing requests to multiple upstream LLM providers. Key capabilities:

- **Canonical Intermediate Format (CIF)** вЂ” normalizes all requests/responses across providers
- **Virtual Models** вЂ” abstract model aliases mapped to upstream providers with load balancing
- **Provider Failover** вЂ” priority-ordered candidate resolution with automatic fallback
- **Streaming (SSE)** вЂ” with resilience fallback (stream в†’ non-stream retry в†’ re-stream)
- **Admin UI + CLI** вЂ” provider management, virtual models, logs, configuration
- **SQLite Persistence** вЂ” provider configs, tokens, model state, virtual models stored locally

## 7. Model Routing

### 7.1 Model Resolution вЂ” `internal/lib/modelrouting/modelrouting.go`

**`ResolveProvidersForModel(requestedModel, normalizedModel, providerID, cache)`**:
1. If `providerID` is non-empty, filter active providers to that specific instance
2. If exactly one provider is pinned, synthesize a passthrough model entry (skip /models check)
3. Otherwise, fetch models from all active providers via `GetEnabledModelsByProvider()`
4. Find the selected model by matching `requestedModel` or `normalizedModel`
5. Filter candidate providers that have the model, respecting API modes
6. Sort candidates by priority (from DB `provider_instances.priority`)

**`ParseProviderPrefix(modelName)`** вЂ” splits `"instanceID/modelID"` в†’ `(instanceID, modelID)`. Used for provider-pinned requests.

**`NormalizeModelName(modelName)`** вЂ” lowercases and maps aliases:
- `gpt-4` в†’ `gpt-4o`
- `gpt-3.5-turbo` в†’ `gpt-4o-mini`
- `claude-3-sonnet` в†’ `claude-3-5-sonnet-20241022`
- `haiku`, `haiku-4.5`, `claude-haiku-4.5` в†’ `claude-haiku-4.5`
- `sonnet`, `sonnet-4`, `claude-sonnet-4` в†’ `claude-sonnet-4`
- `sonnet-4.5` в†’ `claude-sonnet-4.5`
- `sonnet-4.6` в†’ `claude-sonnet-4.6`
- `opus`, `opus-4.6` в†’ `claude-opus-4.6`

**`ModelCache`** вЂ” in-memory cache with RWMutex, maps instanceID в†’ ModelsResponse.

### 7.2 Virtual Model Routing вЂ” `internal/lib/virtualmodelrouting/virtualmodelrouting.go`

**`SelectUpstream(upstreams, strategy, virtualModelID)`** вЂ” picks one upstream.

**`OrderUpstreams(upstreams, strategy, virtualModelID)`** вЂ” returns ordered list for failover:

| Strategy | Behavior |
|----------|----------|
| `round-robin` | Per-model cursor, rotates through upstreams |
| `random` | Random selection with `rand/v2.IntN()` |
| `priority` | Returns as-is (sorted by priority in DB) |
| `weighted` | Weighted random selection, weights default to 1 |

### 7.3 Route-Level Resolution вЂ” `internal/routes/model_resolution.go`

**`resolveRequestedModels(requestID, requestedModel)`**:
1. Check for provider prefix (`instanceID/modelID` format) вЂ” if present, pin to that provider
2. Resolve prefix via `resolveProviderPrefix()` (exact instance ID match в†’ case-insensitive subtitle match)
3. Look up virtual model by normalized name
4. If virtual model found and enabled, load upstreams and order by strategy
5. Return ordered list of `resolvedModelAttempt{RequestedModel, NormalizedModel, ProviderID}`

## 9. Request Flow Details

### 9.1 Inbound Processing

1. **Auth**: Bearer token or `x-api-key` header (constant-time comparison via `internal/server/auth.go`)
2. **Ingestion**: Parse request body into `cif.CanonicalRequest`
   - `ingestion/from_chat_completion.go` вЂ” OpenAI `/v1/chat/completions` shape
   - `ingestion/from_responses.go` вЂ” OpenAI `/v1/responses` shape
   - `ingestion/from_anthropic.go` вЂ” Anthropic `/v1/messages` shape
3. **Normalization**: `prepareCanonicalRequest()` in `route_logging.go` вЂ” captures incoming headers, sets inbound API shape, logs request

### 9.2 Model Routing

1. `resolveRequestedModels()` in `model_resolution.go` вЂ” handles provider prefixes and virtual models
2. `modelrouting.ResolveProvidersForModel()` вЂ” maps model to candidate providers
3. Provider prefix resolution: exact instance ID в†’ case-insensitive subtitle match
4. Virtual model upstream ordering by strategy (round-robin, random, priority, weighted)

### 9.3 Provider Execution

1. `registry.GetAdapter()` вЂ” get provider adapter
2. `adapter.RemapModel()` вЂ” remap model name for upstream
3. **Copilot compat** (`copilot_compat.go`): sets `ForceChatCompletions`, `DisableAuthRetry`, `DisableStreamingFallback` for Copilot provider
4. **Upstream API detection** (`request_extensions.go`): `detectUpstreamAPI()` checks if adapter implements `upstreamAPIReporter`, falls back to `providermodels.UpstreamAPI()`
5. **Streaming**: `ExecuteStream()` returns `<-chan cif.CIFStreamEvent`
6. **Non-streaming**: `Execute()` returns `*cif.CanonicalResponse`
7. **Fallback**: If stream fails before any data and `allowStreamingFallback()` is true, retry with `Execute()` and re-stream result locally

### 9.4 Outbound Serialization

1. `serialization/to_openai.go` вЂ” CIF в†’ OpenAI SSE format (chat completions)
2. `serialization/to_responses.go` вЂ” CIF в†’ Responses API SSE format
3. `serialization/to_anthropic.go` вЂ” CIF в†’ Anthropic format

### 9.5 Tool Loop Logging вЂ” `internal/routes/tool_loop_logging.go`

Debug-level logging for tool call/result loops:
- `toolLoopCallTracker` вЂ” observes streaming deltas, builds tool call log entries
- Extracts tool results from CIF requests and raw Anthropic payloads
- Agent tool guardrail: detects missing tool_result pairs, client execution failures
- All values truncated to 400 chars

## 14. Key Implementation Notes

- **CIF is the core abstraction** вЂ” all request/response normalization flows through it; content parts, messages, tools, streaming events all canonicalized
- **ProviderAdapter** centralizes Execute/ExecuteStream and model remapping; each provider implements its own adapter
- **Upstream API shape** is determined declaratively via `providermodels` table, with per-instance overrides via `api_format` config
- **Streaming resilience** вЂ” automatic fallback from SSE to non-streaming when upstream fails before any data; can be disabled per-provider
- **Virtual models** вЂ” admin-controlled aliasing with four load-balancing strategies (round-robin, random, priority, weighted); provider display prefixes for self-describing upstream model IDs
- **Provider prefix routing** вЂ” `instanceID/modelID` syntax pins requests to specific providers; resolved via exact ID or case-insensitive subtitle match
- **Pure Go SQLite** вЂ” no CGO dependency, WAL mode, 5 schema migrations applied idempotently
- **Rate limiting** вЂ” configurable interval with optional wait-on-limit behavior
- **Manual approval** вЂ” supported via `lib/approval/`
- **Config editing** вЂ” disabled by default, requires `--enable-config-edit`
- **OAuth flows** вЂ” GitHub Copilot (device code with polling), Antigravity (Google OAuth 2.0 with server-side state management and frontend polling)
- **SSRF protection** вЂ” endpoint validation blocks loopback, private, link-local IPs unless explicitly allowed
- **Tool loop logging** вЂ” debug-level observability for tool call/result pairs, agent tool guardrail for detecting missing results and client failures
- **Model cache** вЂ” 24h TTL per-provider model list cache in SQLite, with force-refresh support
- **Model name normalization** вЂ” aliases for common model names (gpt-4в†’gpt-4o, haikuв†’claude-haiku-4.5, etc.)
- **Copilot single-upstream mode** вЂ” forces chat completions, disables auth retry and streaming fallback for Copilot provider
