# OmniLLM — Codebase Snapshot

> Captured: 2026-04-27 | Branch: master | Latest commit: 38108f2

## 1. What This Project Does

**OmniLLM** is an enterprise-ready multi-provider LLM gateway. It exposes a single API surface compatible with both **OpenAI** and **Anthropic** endpoints while routing requests to multiple upstream LLM providers. Key capabilities:

- **Canonical Intermediate Format (CIF)** — normalizes all requests/responses across providers
- **Virtual Models** — abstract model aliases mapped to upstream providers with load balancing
- **Provider Failover** — priority-ordered candidate resolution with automatic fallback
- **Streaming (SSE)** — with resilience fallback (stream → non-stream retry → re-stream)
- **Admin UI + CLI** — provider management, virtual models, logs, configuration
- **SQLite Persistence** — provider configs, tokens, model state, virtual models stored locally

## 2. Architecture Overview

```
Client Request (OpenAI or Anthropic shape)
  │
  ▼
Auth Middleware (Bearer token / x-api-key)
  │
  ▼
Ingestion Layer ──► cif.CanonicalRequest
  │                   (from_chat_completion.go)
  │                   (from_responses.go)
  │                   (from_anthropic.go)
  ▼
Model Routing ──► CandidateProviders (priority order)
  │                 (modelrouting/ + virtualmodelrouting/)
  ▼
Provider Adapter ──► Execute / ExecuteStream
  │                    (copilot, alibaba, azure, google, kimi,
  │                     codex, antigravity, openaicompat, generic)
  ▼
Serialization ──► cif.CanonicalResponse / Stream Events
  │                 (to_openai.go, to_responses.go, to_anthropic.go)
  ▼
Client Response (matching inbound API shape)
```

## 4. Core Types & Interfaces

### 4.1 CIF (Canonical Intermediate Format) — `internal/cif/types.go`

**CanonicalRequest** fields:
- `Model`, `SystemPrompt`, `Messages []CIFMessage`
- `Tools []CIFTool`, `ToolChoice`
- `Temperature`, `TopP`, `MaxTokens`, `Stop []string`
- `Stream`, `UserID`, `IncomingHeaders`
- `ResponseFormat`, `PreviousResponseID`
- `Extensions` (ThinkingBudgetTokens, InboundAPIShape, etc.)

**CIFMessage types**: CIFSystemMessage, CIFUserMessage, CIFAssistantMessage

**CIFContentPart** interface implementations:
- `CIFTextPart` — text content
- `CIFImagePart` — image content
- `CIFThinkingPart` — reasoning/thinking content
- `CIFToolCallPart` — tool invocation
- `CIFToolResultPart` — tool result (can contain nested parts)

**CanonicalResponse** fields:
- `Model`, `Content []CIFContentPart`, `Usage CIFUsage`
- `StopReason` (CIFStopReason constants)

**Streaming Events**:
- `CIFStreamStart`, `CIFContentDelta`, `CIFContentBlockStop`
- `CIFStreamEnd`, `CIFStreamError`

### 4.2 Provider Interface — `internal/providers/types/types.go`

```go
type ProviderAdapter interface {
    GetProvider() Provider
    Execute(ctx context.Context, req *cif.CanonicalRequest) (*cif.CanonicalResponse, error)
    ExecuteStream(ctx context.Context, req *cif.CanonicalRequest) (<-chan cif.StreamEvent, error)
    RemapModel(canonicalModel string) string
}
```

**Provider** interface:
- `GetID()`, `GetInstanceID()`, `GetName()`, `SetName()`
- `SetupAuth(AuthOptions)`, `GetToken()`, `RefreshToken()`
- `GetBaseURL()`, `GetHeaders()`, `GetModels()`
- `GetUsage()`, `GetAdapter()`

**AuthOptions** — supports APIKey, Token, Region, Models (JSON-encoded), Deployments, APIVersion, etc. with flexible JSON unmarshaling (camelCase/snake_case).

### 4.3 Database Records — `internal/database/types.go`

- `ProviderInstanceRecord` — provider instance metadata
- `TokenRecord` — OAuth/API token storage
- `ProviderConfigRecord` — provider configuration
- `ProviderModelStateRecord` — provider model state
- `ModelConfigRecord`, `ProviderModelsCacheRecord` — model caching
- `VirtualModelRecord`, `VirtualModelUpstreamRecord` — virtual models
- `ChatSessionRecord`, `ChatMessageRecord` — chat history
- `LbStrategy` constants — load balancing strategies

## 8. Request Flow Details

### 8.1 Inbound Processing

1. **Auth**: Bearer token or `x-api-key` header (constant-time comparison)
2. **Ingestion**: Parse request body into `cif.CanonicalRequest`
   - `ingestion/from_chat_completion.go` — OpenAI shape
   - `ingestion/from_responses.go` — Responses API shape
   - `ingestion/from_anthropic.go` — Anthropic shape
3. **Normalization**: System prompt handling, header storage, inbound shape tracking

### 8.2 Model Routing

1. `modelrouting.Resolve()` — maps requested model to candidate providers
2. Model cache (`ProviderModelsCacheRecord`) for fast lookups
3. Virtual model routing (`virtualmodelrouting/`) for abstract model aliases
4. Load balancing: weighted, priority, or round-robin strategies

### 8.3 Provider Execution

1. `registry.GetAdapter()` — get provider adapter
2. `adapter.RemapModel()` — remap model name for upstream
3. **Streaming**: `ExecuteStream()` returns `<-chan cif.StreamEvent`
4. **Non-streaming**: `Execute()` returns `*cif.CanonicalResponse`
5. **Fallback**: If stream fails before any data, retry with `Execute()` and re-stream

### 8.4 Outbound Serialization

1. `serialization/to_openai.go` — CIF → OpenAI SSE format
2. `serialization/to_responses.go` — CIF → Responses API SSE format
3. `serialization/to_anthropic.go` — CIF → Anthropic format

## 13. Key Implementation Notes

- **CIF is the core abstraction** — all request/response normalization flows through it
- **ProviderAdapter** centralizes Execute/ExecuteStream and model remapping
- **Streaming resilience** — automatic fallback from SSE to non-streaming when upstream fails before data
- **Virtual models** — admin-controlled aliasing with weighted/priority routing, provider display prefixes
- **Pure Go SQLite** — no CGO dependency, portable across platforms
- **Rate limiting** — implemented via `lib/ratelimit/`, used by handlers
- **Manual approval** — supported via `lib/approval/`
- **Config editing** — disabled by default, requires `--enable-config-edit`
- **OAuth flows** — GitHub Copilot (device code), Antigravity (Google OAuth)
