OmniLLM architecture overview

One gateway, one coding client, one shared control plane.

OmniLLM and OmniProxy expose the same server-backed gateway surface, while OmniCode is the coding-focused CLI that consumes that server over HTTP. This page shows the three layers as a single visual flow.

Gateway: omnillm / omniproxy
Coding CLI: omnicode
Shared core: CIF, providers, sessions, admin APIs

Client binary

omnicode

OmniCode is a CLI client that connects to the running gateway and uses the shared API surface.

Entry layer cmd/omnicode/main.go → internal/omnicode/app.go
  • Runs chat and agent loops
  • Creates or reuses sessions
  • Consumes the server, does not replace it
Start point Launch omnillm or omniproxy first. They expose the backend contract that all clients consume.
Interaction point OmniCode talks to the server over HTTP using the same API key and session flow as other clients.
Why it matters The server remains reusable while the CLI stays specialized for coding workflows and agent behavior.
One sentence OmniLLM/OmniProxy are the platform; OmniCode is the coding client on top of that platform.
