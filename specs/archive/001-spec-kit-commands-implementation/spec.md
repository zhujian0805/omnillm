# Spec: 001 Spec Kit Commands Implementation

**Created**: 2026-05-12T04:16:25Z

## Overview

Show all available commands documented in https://github.com/github/spec-kit and ensure those commands are implemented in the product. OmniCode should expose the core Spec Kit workflow commands in `/spec` help, slash command picker suggestions, and the `spec` skill tool set, while keeping existing `spec_*` tool aliases usable.

## User Stories

### US1 - Discover supported Spec Kit commands (P1)

As an OmniCode user entering spec-driven mode, I want `/spec` and `/spec mode spec-kit` to show the current Spec Kit command surface so I can choose the correct workflow step.

**Why P1**: The feature request explicitly asks to show all commands from the Spec Kit repository.

#### Acceptance Scenarios

- **Given** the user is in chat or agent mode, **when** they type `/spec`, **then** the response shows spec-kit as a workflow and lists the supported `/speckit.*` core commands.
- **Given** the user chooses `/spec mode spec-kit`, **when** the workflow summary is printed, **then** it includes `/speckit.constitution`, `/speckit.specify`, `/speckit.clarify`, `/speckit.plan`, `/speckit.tasks`, `/speckit.analyze`, `/speckit.implement`, and `/speckit.checklist`.
- **Given** the user opens the slash command picker and types `/spec ` or `/speckit`, **when** suggestions are shown, **then** the available Spec Kit commands are included.

### US2 - Invoke Spec Kit-compatible agent tools (P1)

As an agent using the loaded `spec` skill, I want tools named after Spec Kit phases so I can execute the documented workflow directly rather than translating to legacy `spec_*` names.

**Why P1**: The request says the commands should be implemented, not just documented.

#### Acceptance Scenarios

- **Given** the agent loads the `spec` skill, **when** tool definitions are listed, **then** Spec Kit-compatible tools are available for constitution, specify, clarify, plan, tasks, analyze, implement, and checklist.
- **Given** the agent calls `speckit_specify` with a feature description, **when** the tool runs, **then** it creates or updates a numbered `specs/<N>-<slug>/spec.md` artifact.
- **Given** the agent calls planning/task commands, **when** required prior artifacts are missing, **then** the command returns a clear actionable error rather than silently succeeding.

### US3 - Preserve existing spec workflow compatibility (P2)

As an existing OmniCode user, I want `spec_init`, `spec_write`, `spec_read`, `spec_plan`, `spec_tasks`, and `spec_status` to continue working so current sessions and scripts do not break.

**Why P2**: Backward compatibility reduces migration risk.

#### Acceptance Scenarios

- **Given** the `spec` skill is loaded, **when** definitions are listed, **then** legacy `spec_*` tools are still present.
- **Given** existing tests invoke `spec_init -> spec_write -> spec_plan -> spec_tasks`, **when** they run, **then** behavior remains unchanged.

## Functional Requirements

- **FR-001 [US1]** The system SHALL list the core Spec Kit commands documented in the upstream README and templates: `/speckit.constitution`, `/speckit.specify`, `/speckit.clarify`, `/speckit.plan`, `/speckit.tasks`, `/speckit.analyze`, `/speckit.implement`, and `/speckit.checklist`.
- **FR-002 [US1]** The system SHALL expose these commands in `/spec` help text and `/spec mode spec-kit` workflow summaries.
- **FR-003 [US1]** The slash command picker SHALL include discoverable entries for `/speckit.*` and `/spec`-scoped equivalents.
- **FR-004 [US2]** The `spec` skill SHALL register callable agent tools for the Spec Kit-compatible phases.
- **FR-005 [US2]** Spec Kit-compatible tools SHALL create or validate the expected artifacts (`memory/constitution.md`, `spec.md`, `plan.md`, `tasks.md`, `checklists/*.md`, and analysis output) according to the phase.
- **FR-006 [US2]** Implement/analyze commands SHALL be non-destructive unless prerequisites are satisfied and SHALL report actionable status.
- **FR-007 [US3]** Legacy `spec_*` tools SHALL remain registered and continue to pass existing behavior tests.

## Key Entities

- **SpecKitCommand**: command name, summary, prerequisites, output artifact, and implementation status.
- **SpecArtifact**: markdown file or directory generated by a command, including `spec.md`, `plan.md`, `tasks.md`, checklist files, constitution, and analysis report.
- **SpecSkillTool**: agent-callable tool registered under the `spec` skill.

## Edge Cases

- **EC-001**: If a command requiring an active spec is called before any spec exists, the system should return a clear message instructing the user to run specify/init first.
- **EC-002**: If `/speckit.*` is typed in the REPL, the system should guide the user toward `/spec` or agent tools rather than treating it as an unknown opaque command.
- **EC-003**: If upstream adds extension commands, the product should focus on core commands and avoid claiming community extension support unless implemented.
